Security Checklist

28 Dec 2020

This isn't an exhaustive list. It's only a summary of my current knowledge.
  • Don't use SameSite=none on user-facing domains
  • Don't use GET, HEAD, or OPTIONS to perform authenticated state changes
  • Use secure cookies
  • Don't use method-override headers
  • Use PATCH, PUT, or DELETE instead of POST wherever it makes sense
  • Don't create any kind of "/csrf" route where you can access the CSRF token directly
  • Don't use url parameters for authenticated POST requests
  • Salt your CSRF tokens
  • Use a constant-time or random-time algorithm to verify tokens
  • Use CSRF tokens not only for authenticated state changes, but also for requests that return sensitive data
  • In the double-submit cookie pattern, make the cookie be a signature
  • Check the origin of your requests
  • Put CSRF protection on login pages
  • Start a new session after login
  • If attackers are able to overwrite cookies, use the synchronizer token pattern instead of double-submit cookies
  • Use x-frame-options to prevent clickjacking
  • Don't store session IDs in urls
  • Use rate-limiting
  • Use latency examination to detect MITM attacks
  • Use HSTS
  • Use a certificate transparency monitor
  • Prefer host-only cookies
  • Use the "__Secure-" or "__Host-" cookie prefixes
  • Disable TLS compression
  • Disable SPDY header compression
  • Use includeSubdomains for HSTS wherever it makes sense
  • Use input validation
  • Use blacklists or whitelists on user-controllable server requests
  • Prefer whitelists to blacklists
  • Automatically redirect from http to https
  • Use a Content Security Policy
  • Use the Helmet library for Express.js, or equivalent protections
8 Jan 2021 update: this checklist is too specific to sessions. Just use the OWASP Cheat Sheet Series.
